# 일반 사용자 배포 전략

"개발자가 아닌 사람이 이 SRT/KTX 예약 매크로를 어떻게 쓰게 할 것인가"에 대한 분석.
코드가 아니라 배포 형태에 대한 문서다.

## 1. 전제와 제약

현재 제공 형태는 두 가지다.

- **macOS 데스크톱 앱** — PyInstaller로 빌드한 `.app` (tkinter GUI, `dist/기차예약매크로.app`).
  개발자 본인은 `기차예약매크로.command` 더블클릭으로 소스를 직접 실행 중.
- **웹 버전** — `docker compose up`으로 backend(FastAPI, :8000) + frontend(Next.js, :3000)를
  로컬에서 띄우는 구조 ([web.md](web.md) 참고).

코어는 SRTrain/korail2 파이썬 라이브러리로, SRT/코레일 모바일 앱의 비공개 API를
리버스 엔지니어링한 HTTP 클라이언트다. 여기서 바꿀 수 없는 제약이 나온다.

1. **클라우드 호스팅 불가.** 모든 사용자의 예매 요청이 서버 IP 몇 개로 집중되면
   SRT의 이상 접근 차단("Your IP Address Blocked due to abnormal access")에 확정적으로
   걸리고, 한 번 차단되면 전체 사용자가 동시에 죽는다. 데이터센터 IP 대역은 그 자체로
   차단 대상이다.
2. **법적 리스크.** 불특정 다수에게 자동 예매를 서비스로 제공하는 것은 개인 사용과
   차원이 다르다(철도사업법상 부정구매, 업무방해 소지). 사용자 계정정보를 수탁하는
   책임도 생긴다.
3. 따라서 **"실행은 사용자 로컬, 우리는 소프트웨어만 배포"** 가 확정 전제다.
   요청이 각 사용자의 가정용 IP에서 나가야 차단 리스크가 분산되고, 계정정보도
   사용자 기기를 벗어나지 않는다.

향후 계획: SRTrain을 JS로 포팅해 npm 패키지화하고 백엔드를 Node로 교체한다.
프론트는 `web/lib/api.ts`의 REST API 계약을 그대로 유지하므로 재작성이 필요 없다.

## 2. 배포 옵션별 분석

### a. macOS .app / Windows .exe 직접 배포

현재 `.app`이 이미 존재한다 (`TrainMacro.spec`, `codesign_identity=None` — 미서명).

- **대상**: 앱 설치는 할 줄 아는 일반 사용자.
- **설치 과정**: 다운로드 → 압축 해제 → 실행. 다만 미서명 상태에서는 여기가 끝이 아니다.
  - **macOS Gatekeeper**: 서명/공증(notarization) 없는 앱은 "확인되지 않은 개발자"
    경고로 실행이 차단된다. 우회하려면 우클릭→열기, 최신 macOS에서는
    시스템 설정→개인정보 보호 및 보안에서 수동 허용까지 필요. 일반 사용자에게는
    "악성코드 아님?"이라는 인상을 주는 큰 이탈 지점이다. 해결하려면
    Apple Developer Program(연 $99) 가입 + notarize 파이프라인이 필요.
  - **Windows SmartScreen**: 서명 없는 exe는 "PC를 보호했습니다" 경고. EV 코드서명
    인증서(연 수십만 원)를 사도 평판이 쌓이기 전에는 경고가 뜰 수 있다.
    Windows용은 빌드 자체도 없어 Windows 머신/CI에서 PyInstaller 빌드를 새로 만들어야 한다.
- **장점**: 설치가 개념적으로 가장 단순. Docker/Node 등 선행 설치 없음. 오프라인 배포 가능.
- **단점**: 서명 비용, OS별 빌드 유지(macOS arm64/x64, Windows), tkinter GUI 품질 한계,
  자동 업데이트 없음(버그 수정마다 재배포·재설치).
- **추가 작업**: 코드서명/공증 파이프라인, Windows 빌드, 업데이트 안내 채널.
- **난이도**: 배포자 중간~높음 / 사용자 낮음(서명했을 때 기준).

### b. docker-compose (현재 웹 버전)

- **대상**: 개발자이거나 최소한 터미널을 무서워하지 않는 사용자.
- **설치 과정**: Docker Desktop 설치 → 소스 다운로드 → 터미널에서
  `docker compose up -d --build` → 브라우저로 localhost:3000 접속. 4단계지만
  각 단계가 일반 사용자에게 전부 낯설다.
- **장점**: 환경 격리로 "내 컴퓨터에선 안 돼요"가 없음. 이미 동작하는 자산.
  UI가 브라우저라 데스크톱 tkinter보다 낫다.
- **단점**: Docker Desktop 자체가 결정적 장벽 — 수 GB 설치, 계정/라이선스 안내,
  가상화 켜기(특히 Windows의 WSL2). "매크로 하나 쓰려고 Docker를 깐다"는 요구는
  비개발자에게 사실상 불가능한 수준. 첫 빌드도 수 분 걸린다.
- **추가 작업**: 거의 없음(이미 완성). 이미지를 미리 빌드해 레지스트리에 올리면
  `--build`는 생략 가능하나 본질적 장벽은 그대로.
- **난이도**: 배포자 낮음(완료 상태) / 사용자 높음.

### c. npx 원커맨드 실행 (Node 백엔드 포팅 완료 후)

`npx srt-macro` 한 줄로 로컬 서버를 띄우고 브라우저를 자동으로 여는 방식.
Next.js 프론트는 정적 빌드해 패키지에 동봉하고, Node 백엔드가 정적 파일과
REST API를 같은 포트에서 서빙하면 된다.

- **대상**: 터미널에 한 줄은 붙여넣을 수 있는 사용자 (블로그 안내글 따라하기 수준).
- **설치 과정**: Node.js 설치(설치 마법사, 1회) → 터미널에 `npx srt-macro` 붙여넣기 → 끝.
  실질 2단계.
- **장점**: 업데이트가 자동(npx가 항상 최신 버전을 받음 — 재배포·재설치 개념이 없다).
  코드서명 문제 없음. OS별 빌드 없음(순수 JS라면). UI는 기존 웹 프론트 재사용.
  배포 채널이 npm publish 하나로 끝난다.
- **단점**: Node.js 선행 설치가 필요 — Docker보다는 훨씬 가볍지만(공식 인스톨러
  더블클릭) 여전히 비개발자에게는 한 단계다. "터미널을 연다"는 행위 자체가
  일부 사용자에겐 장벽. SRTrain JS 포팅이 선행 조건.
- **추가 작업**: SRTrain/korail2 JS 포팅(핵심 공수), Node 백엔드 작성(api.ts 계약 준수),
  프론트 정적 빌드 동봉 + 브라우저 자동 오픈, npm 패키지화.
- **난이도**: 배포자 높음(포팅) / 사용자 낮음~중간.

### d. Electron/Tauri 데스크톱 앱으로 웹 UI 래핑

기존 웹 프론트 + (포팅 후) Node 백엔드를 하나의 데스크톱 앱으로 묶는 방식.
Electron은 Node 내장이라 c의 결과물을 거의 그대로 품을 수 있다. Tauri는 바이너리가
작지만 백엔드가 Rust여서 사이드카(별도 프로세스)로 Node를 끼워야 해 이점이 줄어든다.

- **대상**: 가장 넓은 일반 사용자. 터미널조차 안 열게 하고 싶을 때의 종착지.
- **설치 과정**: dmg/설치파일 다운로드 → 실행. 1~2단계.
- **장점**: 사용자 경험 최상. 선행 설치 없음(Node 런타임 내장). 자동 업데이트
  프레임워크(electron-updater 등) 사용 가능. UI는 웹 프론트 그대로.
- **단점**: a와 동일한 코드서명/공증 문제가 그대로 돌아온다(미서명이면 Gatekeeper/
  SmartScreen 경고). Electron은 배포 파일이 100MB+. OS별 빌드/릴리스 파이프라인 유지 필요.
  c가 끝나야 시작할 수 있는 옵션.
- **추가 작업**: c의 전부 + Electron 셸, 서명/공증, OS별 CI 빌드.
- **난이도**: 배포자 높음 / 사용자 최저.

### e. (반면교사) 호스팅 SaaS

"사이트 가입하고 클릭 한 번"이 사용자 경험으로는 최고지만, 이 프로젝트에서는 성립하지 않는다.
모든 예매 트래픽이 서버 IP로 집중되는 순간 SRT IP 차단으로 전 사용자 동시 장애가
확정이고, 데이터센터 IP는 애초에 차단 대상이다. 여기에 자동 예매의 영업 제공이라는
법적 리스크와 타인 계정정보 보관 책임까지 얹힌다. 기술적으로 프록시 분산 등으로
회피를 시도하는 것 자체가 리스크를 키우는 방향이므로 검토 대상에서 제외한다.

## 3. 옵션 비교표

| 옵션 | 사용자 설치 난이도 | 개발 비용 | 유지보수 | 추천 대상 |
|---|---|---|---|---|
| a. .app/.exe 직접 배포 | 낮음 (서명 시) / 중간 (미서명) | 중간 (서명·OS별 빌드) | 높음 (버전마다 재배포) | 지인 소수, 지금 당장 |
| b. docker-compose | **높음** (Docker 필수) | 없음 (완료) | 낮음 | 개발자·파워유저 |
| c. npx 원커맨드 | 낮음~중간 (Node 1회 설치) | 높음 (JS 포팅) | **낮음** (npm publish로 끝) | 안내글 따라할 수 있는 일반 사용자 |
| d. Electron/Tauri | **최저** | 최고 (c + 셸 + 서명) | 중간 (OS별 릴리스) | 최대 다수, 최종 형태 |
| e. 호스팅 SaaS | — | — | — | **불가** (IP 차단 + 법적 리스크) |

## 4. 권장 로드맵

### 단기 — 현재 자산으로

1. **지인 소수에게는 미서명 `.app` + Gatekeeper 우회 안내문**으로 배포한다.
   이미 빌드가 있고, "우클릭→열기" 스크린샷 안내면 소수 대상으로는 충분하다.
   Apple Developer 서명은 배포 대상이 실제로 늘어나기 전까지는 비용 대비 이르다.
2. **터미널이 가능한 사용자에게는 docker-compose 버전**을 web.md 기반 안내로 제공한다.
   추가 개발이 0이고, 웹 UI가 데스크톱보다 낫다.
3. 이 단계에서 새 배포 채널에 투자하지 않는다. 근거: 어차피 Node 포팅으로 백엔드가
   교체될 예정이라, 파이썬 스택 기준의 배포 개선(예: PyInstaller Windows 빌드,
   파이썬 원파일 서버)은 곧 버려질 공수다.

### 중기 — npm 포팅 완료 후

4. **`npx srt-macro`를 표준 배포 채널로 삼는다.** 근거: (1) 코드서명 문제를 완전히
   우회하면서 사용자 난이도는 "Node 설치 + 한 줄 복붙"까지 내려간다. (2) 유지보수가
   `npm publish` 하나로 끝나고 사용자는 항상 최신 버전을 받는다 — 비공개 API 대응처럼
   잦은 패치가 필요한 이 프로젝트 특성에 결정적이다. (3) 프론트는 api.ts 계약 그대로
   재사용한다.
5. **수요가 검증되면 그때 Electron 래핑 + 코드서명을 얹는다.** Electron은 c의 결과물을
   거의 그대로 품으므로 c가 자연스러운 선행 단계다. 서명 비용(연 $99+)과 OS별 릴리스
   파이프라인은 "터미널도 못 여는 사용자"가 실제로 유의미하게 확인된 뒤에 지불한다.

요약: **b(현재) → c(표준) → d(수요 시)**. a는 단기 임시 채널, e는 불가.

## 5. 공통 주의사항 (어떤 배포 형태든 안내에 포함할 것)

- **개인 사용 목적 한정.** 타인 대행 예매·유상 제공은 하지 않는다.
- **계정 정보는 로컬에만 저장된다.** (`config.*.json` — 사용자 기기 밖으로 나가지 않음.)
- **결제는 하지 않는다.** 예매 성공 후 결제는 SRT/코레일 공식 앱·웹에서 결제 기한 내에
  직접 해야 하며, 매크로는 좌석 확보까지만 담당한다.
- 매크로가 도는 동안에는 컴퓨터가 켜져 있어야 한다.
- 과도한 조회 간격 설정은 계정/IP 차단으로 이어질 수 있다.
